Minutes of the 3-Dec-2002 CGS-WG telephone conference

December 3, 2002, (PG)

The 5th CGS-WG telecon was held on December 3, 2002 (8AM Pacific time). Attendees were:

Viktor Mihajlovski (IBM Germany)
Ellen Stokes (IBM)
Peter Gietz (DAASI)
Larry Flon (USC/ISI) (only a short while)

Agenda

1. Agenda bashing
2. Changes in the draft (Ellen)
3. Open questions:
   - separate class for statistics?
   - additional types of managed properties
   - BatchJobGroup vs. work flow
   - properties of BatchService
   - other comments on the new draft
4. any other business

  • 1. Agenda bhashing
    No changes to the agenda were made
  • 2. Changes in the draft
    Ellen reported working on changes in the Visio:
    • Two new attributes:
      1.) Scheduledpriority (priority set by the scheduler which may be differetn than the the priority set by the user)
      2.) JobWallTimeUsed (number of seconds that a job queued)
    • Two new associations between system and batchJob:
      1.) Execution candidate (relationship between a batch job and one or more nodes capable of running the job)
      2.) ExecutionTarget (showing that the job is running on one or more nodes)

      Associations to System are inherited to computersystem and cluster. It was noted that the cluster modell is being changed and that
      we should keep an eye on these changes in respect to this association. 
    • Deleted association between batchQueue and batchSAP because it seems to be unneccesary

  • 3. Open questions: 
    • separate class for statistics?
      all agreed that this makes sense
    • additional types of managed properties:
      a. description of task to be executed: script/command/program, parameters, recovery options, managementof output
          Discussed the possibilities of one freeform descrition attribute vs. one attribute with compound syntax vs.
          several attributes for this. No final decision. Ellen will get back to the person who introduced this idea.
      b. userid of the user to impersonate for job execution
          All agreed that such a pointer to user data will be needed. In future the user and security classes will
          be modified segregating the White Pages information from the security information. We agreed that in this
          context the security information is more relevant. Again something to watch. Another input was that the
          proxy-certificate impersonation defined in GSI should be taken into account.
      c. failure/success condition on return code
         This is not needed to be added since this is already in the abstract class job from which batchJob is derived
      d. recovery options: stop, re-run, etc
          All agreed that this is a good idea and needs to be included. It should be a ENUM of about 6 reasonable
          options. Ellen will again check back with the person who ntroduced this idea.

      In general it was remarked that such discussions should be taken to the list at first, before discussing them
      in the telcon

    • BatchJobGroup vs. work flow
      Is there a simplified concept for grouping jobs, e.g. for ordering, without having to go into the ratholes of workflow modelling? The idea is to start
      and experiment with something simple before getting to the work flow stuff. Could the Unix pipe model help? No that is rather implementational stuff. Decision was to take this to the list. 
    • properties of BatchService
      Again this should be taken to the list three people on the phone are too few to get decisions on this. We need input about the actual schedulars.
    • other comments on the new draft
      No other comments. 
       
  • Any other business
    After some emails going forth and back Ellen was chosen to be the representative of GGF in the liaison with DMTF. It seems that the batchJob modell
    can be included into CIM 2.8. which will be out as preliminary in spring and thus finalized end of next year.

The next Telecon of the CGS-WG is scheduled for 8AM Pacific on Tuesday 17, 2002.  Dial-in information will be forthcoming. Since Ellen and others will take holidays, we decided that the following Telcon will be January 21, 2003
